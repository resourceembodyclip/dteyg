# WebLink Navigator

WebLink Navigator 是一个面向技术内容聚合与外部资源导航的开源项目，专注于将分散于多源的技术文章、文档与参考链接进行系统性归集与呈现。项目定位于为开发者、技术研究人员以及内容运营团队提供结构化的外链资源管理方案，解决技术信息检索成本高、资源分散且缺乏统一索引的问题。

本项目以移动端优先的架构设计，对每一篇收录的文章进行元数据提取与分类标记，支持按业务领域、技术栈、发布日期等多维度检索。通过对批量化 URL 的整理与展示，WebLink Navigator 形成了一套可扩展的静态站点生成体系，适用于构建企业级技术文档导航站、个人知识库外链库、以及自动化采集内容的中转展示层。

## 功能概览

**结构化外链归集**：将来自同一数据源的大量文章链接按照预设分类进行分组展示，并提供统计计数与最后更新时间标记。

**多维度筛选与检索**：支持按文章 ID、发布时间范围、分类标签进行筛选，内置轻量级全文检索能力，能够快速定位特定技术主题。

**移动端自适应渲染**：前端模板针对手机浏览器进行适配，确保在小屏幕设备上获得良好的阅读与导航体验。

**静态站点生成模式**：提供脚本将原始链接数据编译为纯静态 HTML 文件，无需后端服务即可部署至任意 Web 服务器或 CDN。

**元数据自动补全**：通过可配置的抓取规则，自动从目标页面提取标题、摘要、作者信息，减少手动整理工作量。

**批量导入与校验**：支持以 CSV 或 JSON 格式批量导入链接列表，自动校验 URL 可访问性并标记异常状态。

**标签与分类管理**：内置分类树管理界面，允许运营人员新增、编辑、合并分类节点，适应不同业务场景的导航需求。

**访问统计与热度标记**：记录每个链接的被点击次数，支持按热门程度排序展示，辅助用户发现高价值内容。

## 应用场景

技术文档团队归档外部参考资源：技术写作人员需要将调研过程中参考的数百篇外部文章统一归档，以便后续审校时回溯依据。WebLink Navigator 提供批量导入与分类标记能力，可快速建立按产品模块或技术领域划分的参考索引。

开发者日常阅读与知识管理：开发者在学习新技术或解决特定问题时会浏览大量技术博客与官方文档，使用本系统可以将这些分散的阅读材料集中管理，形成个人技术知识库的外链部分。

内容聚合站点后台数据管理：运营一个技术资讯聚合类站点时，编辑团队每日需要处理数十条外部链接的入库、分类与发布。WebLink Navigator 的后台管理界面与静态导出功能能够适配此类中等规模的内容运营流程。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/weblink-navigator/weblink-navigator.git

# 进入项目目录
cd weblink-navigator

# 安装依赖（使用 npm）
npm install

# 运行开发服务器
npm run dev

# 构建生产环境静态文件
npm run build
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.0.0 或更高版本 | 是 | 运行时环境，用于执行构建脚本与开发服务器 |
| npm 9.0.0 或更高版本 | 是 | 包管理器，用于安装项目依赖 |
| SQLite 3 数据库引擎 | 是 | 用于本地存储链接元数据与分类信息，无需额外安装，由 better-sqlite3 驱动 |
| 内存 512 MB 以上 | 是 | 构建时需加载全部链接数据到内存进行渲染 |
| 磁盘空间 200 MB 以上 | 是 | 存放源码、依赖包及生成的静态文件 |
| 现代浏览器（Chrome 108+ / Firefox 107+ / Safari 16+） | 否 | 仅用于前端预览，生产环境无浏览器要求 |
| Git 2.30 或更高版本 | 否 | 仅当从源码仓库克隆时需要 |
| 网络连接 | 否 | 仅当首次安装依赖或访问外部源时需要 |

## 贡献指南

本项目的贡献流程遵循标准开源协作规范，所有提交需经过代码审查与自动化检查。具体步骤如下：

1. 在 GitHub 上 fork 本仓库至个人账号，并将 fork 后的仓库克隆至本地开发环境。建议在开发前阅读 docs/development/architecture.md 了解整体设计。

2. 创建新的功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式。提交代码前运行 `npm run lint` 和 `npm run test` 确保代码规范与测试通过。

3. 针对新增的导入器、生成器或前端功能，需在 tests/ 目录下补充对应的单元测试或集成测试用例，确保覆盖率不低于 80%。

4. 提交 Pull Request 时，请在描述中清晰说明改动目的、影响范围以及测试验证情况。PR 标题需符合 Conventional Commits 规范。

5. 项目维护者将在 3 个工作日内完成审查，必要时会提出修改意见。合并后代码将自动触发构建流程并更新文档站点。

## 常见问题

Q: 批量导入 180 条链接时，如何处理部分链接无法访问的情况？

A: 导入器中的 url-validator 模块会并发校验所有链接的可访问性，并将失败记录写入 data/import-errors.log 文件。导入完成后控制台会输出统计摘要，包含成功、失败、超时数量。失败记录可在修复后通过 `--retry` 参数单独重试。

Q: 静态站点生成后，如何增量更新而不是全量重新构建？

A: 项目支持基于数据库变更时间的增量构建模式。执行 `npm run build -- --incremental` 时，构建器会比对 data/cache/ 中的元数据缓存时间戳，仅重新渲染发生变动的文章详情页与分类索引页。新增链接会自动触发全量重新生成以保证索引完整性。
