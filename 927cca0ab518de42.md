# LinkVault

LinkVault 是一个面向开发者、技术研究人员与内容创作者的轻量级技术资源外链聚合与导航系统。该项目不提供内容存储，而是通过结构化分类与标准化入口，帮助用户高效检索、整理与复用分散在多个平台的技术文章、文档、工具站与数据源。

项目定位为“技术外链的收藏夹即服务”，目标用户包括日常需要维护大量技术书签的工程师、需要定期查阅行业资料的分析师，以及希望建立团队公共知识库的技术管理者。LinkVault 通过统一的条目格式、可扩展的分类体系与简洁的 Web 界面，将零散的超链接转化为可检索、可分享、可审计的知识资产。

## 功能概览

**批量导入与去重**：支持从 CSV、JSON 及浏览器书签导出文件批量导入链接，并基于 URL 与标题进行自动去重。

**自定义分类标签**：允许用户为每条链接添加多级标签（如 “backend/message-queue”），并支持标签别名与合并。

**全文检索与过滤**：针对链接标题、描述、标签及来源域名构建轻量级倒排索引，支持前缀匹配与模糊搜索。

**定期可用性检测**：内置定时任务，每日对已收录链接进行 HTTP HEAD 请求检测，标记失效链接并发送通知。

**访问统计与热度排序**：记录每条链接的点击次数与最后访问时间，支持按热度、新增时间或随机排序。

**公开只读视图**：可生成团队内共享的只读导航页，支持按标签或分类过滤，无需登录即可浏览。

**数据导入导出**：提供标准 JSON Schema 导出与导入功能，便于迁移、备份或与其他工具集成。

**RESTful API 接口**：提供完整的 CRUD 与检索 API，支持第三方脚本或自动化工作流调用。

## 应用场景

**技术团队内部知识库的入口层**：团队可将日常遇到的优质技术博客、官方文档、调试工具链接统一收录至 LinkVault，并按照 “前端 / 后端 / DevOps / 数据库” 等一级分类组织。新成员入职时，通过只读视图即可快速了解团队常用的外部资源，缩短技术选型与问题排查的调研时间。

**个人开发者的书签管理替代方案**：开发者可将浏览器中积累的大量书签导出后导入 LinkVault，利用标签与检索功能替代浏览器自带的书签管理器。配合可用性检测功能，定期清理失效链接，保持收藏夹的整洁与可用性。

**技术内容聚合站点的数据源**：内容运营人员或技术博主可将 LinkVault 作为底层数据源，通过 API 拉取指定标签下的链接列表，动态生成 “本周推荐阅读” 或 “常用工具合集” 等专题页面，避免重复手动整理。

**离线文档归档前的预处理**：在将外部文档批量下载为离线存档之前，可先将原始 URL 录入 LinkVault，并利用自定义字段记录存档状态、存档路径及版本信息，形成 “原始链接 - 离线副本 - 元数据” 的三层索引结构。

## 快速开始

以下步骤指导您在本地开发环境中启动 LinkVault 服务。

```bash
# 克隆代码仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装依赖（使用 npm）
npm install

# 复制环境变量模板并填写配置
cp .env.example .env

# 初始化数据库（SQLite 默认）
npm run db:migrate

# 导入示例数据（可选）
npm run seed:demo

# 启动开发服务器（默认端口 3000）
npm run dev
```

启动后访问 http://localhost:3000 即可进入 LinkVault 仪表盘。首次使用建议通过 “设置 - 导入” 功能导入您的浏览器书签 HTML 文件进行测试。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x 或 20.x LTS | 运行时环境，推荐使用官方二进制或 nvm 管理 |
| npm | 9.x 或 10.x | 包管理器，随 Node.js 一并安装 |
| SQLite | 3.35 及以上 | 默认嵌入式数据库，无需单独安装，用于开发与小型部署 |
| PostgreSQL | 14.x 及以上（可选） | 生产环境推荐使用，需单独安装并创建数据库 |
| Redis | 7.x（可选） | 用于缓存与队列，提升可用性检测任务的并发性能 |
| 系统内存 | 至少 512 MB | 开发环境最低要求，生产环境建议 2 GB 以上 |

我们欢迎并感谢任何形式的贡献，包括但不限于提交 Issue、改进文档、修复 Bug 或新增功能。请遵循以下步骤：

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。确保您的 Node.js 版本与安装要求中的指定版本一致。

2. 创建新的功能分支，分支命名格式为 `feature/简述功能` 或 `fix/简述修复`。请勿在主分支上直接提交代码。

3. 编写代码时，请遵循项目现有的 ESLint 与 Prettier 配置。所有新增 API 或核心函数需附带单元测试，测试覆盖率不低于 80%。

4. 提交代码前，请运行 `npm run test` 确保所有测试通过，并运行 `npm run lint` 检查代码风格。Commit 信息请使用语义化格式（如 `feat: add batch import from JSON` 或 `fix: correct health check timeout`）。

5. 发起 Pull Request 至主仓库的 `main` 分支，并在描述中清晰说明改动内容、影响范围以及是否涉及破坏性变更。PR 需要至少一位维护者审核后方可合并。

## 常见问题

**Q: LinkVault 是否支持在线编辑链接内容或缓存文章正文？**

A: 不支持。LinkVault 仅存储链接元数据（标题、描述、标签、来源 URL），不缓存或代理文章正文内容。所有内容访问均直接重定向至原始链接。如需全文检索功能，建议搭配第三方搜索引擎或全文索引服务（如 Meilisearch、Elasticsearch）进行二次集成。

**Q: 可用性检测会误报失效吗？**

A: 检测机制默认使用 HEAD 请求并跟随重定向，超时时间设置为 5 秒。对于部分需要 Cookie 或特定 User-Agent 才能正常响应的站点，可能返回非 2xx 状态码。您可以在配置文件中为特定域名设置白名单，跳过检测或自定义请求头。

**Q: 如何从旧版本的 LinkVault 迁移数据？**

A: 项目提供了 `scripts/migrate-v1-to-v2.js` 迁移脚本，支持从旧版 SQLite 数据库导出数据并转换为新版 JSON Schema 格式。迁移前请完整备份数据库文件。详细步骤请参阅 docs/admin/migration.md。
