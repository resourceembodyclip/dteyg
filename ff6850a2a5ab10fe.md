# ResourceHub

ResourceHub 是一个面向开发者、技术研究人员与内容创作者的轻量级技术资源导航与外链聚合平台。该项目旨在解决信息碎片化背景下高质量技术文章、文档与工具链接难以集中发现与管理的问题，通过结构化目录与标签系统，将分散于多个来源的外部技术资源整合为可检索、可分类、可共享的知识库。

项目定位为“技术外链的中央仓库”，不存储任何原文章内容，仅提供元数据索引与跳转服务。目标用户包括需要持续追踪技术动态的工程师、进行竞品分析的产品经理、以及撰写技术周报或学习笔记的在校学生。ResourceHub 提供开箱即用的静态站点生成能力，支持自定义分类与标签过滤，并内置了针对移动端 H5 页面的链接适配处理方案，特别适用于批量收录来自不同域名、不同路径格式的历史技术稿件。

## 功能概览

**批量链接导入与解析** 支持通过 CSV 或 JSON 格式批量导入外部文章 URL，自动提取标题、路径参数与文件扩展名，生成标准化的内部记录。

**智能分类与标签管理** 允许用户为每条外链分配多个自定义标签，并基于 URL 路径模式自动建议分类（如 Article、Tutorial、Specification），便于后续检索。

**响应式链接预览卡片** 在列表视图中为每个外链生成包含来源域名、路径层级和文件类型的预览卡片，无需点击即可快速识别链接性质。

**全文检索与过滤** 内置基于标题关键词与标签组合的检索接口，支持按时间范围、文件大小（若可获取）和来源域名进行多维度筛选。

**外链健康状态检查** 周期性对已收录的 URL 执行 HTTP 头探测，标记返回 4xx 或 5xx 状态的失效链接，并生成失效报告供管理员清理。

**数据导出与分享** 支持将当前筛选结果导出为 Markdown 表格或纯文本列表，方便嵌入技术文档、周报或 Wiki 页面。

## 应用场景

技术团队内部知识库建设 团队技术负责人可将团队内部积累的 H5 技术文章、问题排查记录和架构设计文档的外链统一收录至 ResourceHub，新成员入职时通过标签筛选即可快速获取指定模块的学习资料，无需逐一询问历史链接。

个人开发者技术周报写作 独立博主或技术写手在阅读大量技术博客后，可将有参考价值的文章链接存入 ResourceHub，并按主题（如性能优化、安全漏洞、框架更新）打标。周报写作时直接按标签导出链接列表，大幅减少手动整理 URL 的时间。

技术社区内容运营 开源社区或技术论坛的管理员可使用 ResourceHub 维护“社区推荐阅读”板块，将分布在多个讨论帖中的优质回答或教程外链聚合为单一入口，并通过健康检查功能定期清理已失效的参考资料，保证社区内容质量。

## 快速开始

以下命令演示了从代码仓库克隆、安装依赖到启动开发服务器的完整流程。

```bash
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行成功后，访问控制台输出的本地地址（默认为 http://localhost:3000）即可进入管理界面。首次启动会自动生成示例数据，包含约 50 条模拟外链记录，供测试检索与分类功能。

## 项目结构

```
resourcehub/
├── config/                         # 全局配置文件目录
│   ├── default.yaml                # 默认端口、数据库路径与日志级别
│   └── categories.json             # 预置分类规则（基于 URL 路径正则匹配）
├── src/
│   ├── core/                       # 核心业务逻辑模块
│   │   ├── linkParser.js           # URL 解析器，提取域名、路径段与扩展名
│   │   ├── tagEngine.js            # 标签自动建议引擎，基于关键词词库
│   │   └── healthChecker.js        # 外链健康状态探测与超时重试机制
│   ├── routes/                     # HTTP 路由层，处理 API 请求
│   │   ├── import.js               # 批量导入接口
│   │   ├── search.js               # 检索与过滤接口
│   │   └── export.js               # 数据导出接口（Markdown / CSV）
│   ├── models/                     # 数据模型与 ORM 映射
│   │   ├── Link.js                 # 链接表结构定义（url, title, status, lastChecked）
│   │   └── Tag.js                  # 标签表与链接-标签关联表
│   └── utils/                      # 通用工具函数
│       ├── logger.js               # 结构化日志输出（JSON 格式）
│       └── validator.js            # URL 格式校验与危险协议拦截
├── static/                         # 管理界面静态资源
│   ├── index.html                  # 单页应用入口
│   └── style.css                   # 响应式布局样式表
├── data/                           # 本地数据库文件与种子数据
│   ├── resourcehub.db              # SQLite 数据库（首次启动自动生成）
│   └── seed/                       # 示例数据 JSON 文件
├── tests/                          # 单元测试与集成测试
│   ├── unit/                       # 针对解析器、标签引擎的独立测试
│   └── integration/                # API 端到端测试用例
├── docs/                           # 完整文档目录（见上方文档导航）
├── scripts/                        # 运维辅助脚本
│   ├── backup.sh                   # 数据库定时备份脚本
│   └── migrate.js                  # 数据表结构迁移工具
├── .env.example                    # 环境变量模板（覆盖默认配置）
├── package.json                    # npm 依赖清单与脚本命令
└── README.md                       # 本文件
```

2. 从主分支检出新的功能分支，分支命名格式为 feature/简短描述 或 fix/问题编号。本地开发时请确保所有单元测试通过，并为新增代码补充对应的测试用例。

3. 提交代码前运行 lint 和格式化工具，本项目使用 ESLint 与 Prettier 统一代码风格。提交信息应遵循约定式提交规范，使用 fix:、feat:、docs: 等前缀。

4. 推送分支至远程仓库并创建 Pull Request，描述中需明确关联的 issue 编号、变更内容与测试覆盖情况。PR 至少需要一名维护者审核通过方可合并。

5. 合并后，维护者将同步更新文档中的变更日志，并在下一个版本发布时将其纳入 release note。

## 常见问题

问：导入大量 URL 时页面出现超时或卡顿，应该如何处理？

答：批量导入接口默认单次接受最多 200 条记录。若需导入超过此数量的链接，建议将数据拆分为多个批次，或使用 scripts/ 目录下的命令行导入工具，该工具绕过 HTTP 超时限制并输出进度日志。此外，检查数据库文件是否存储在高速存储介质上，SQLite 在高并发写入时可能产生锁等待。

问：健康检查功能报告大量链接为失效，但浏览器中可正常访问，是什么原因？

答：健康检查默认使用 HEAD 请求且不跟随重定向。部分站点对 HEAD 请求返回 405 或 403，但对 GET 请求正常响应。解决方案为在配置文件中将 checkMethod 改为 GET，并开启 followRedirects 选项。同时检查网络环境是否允许向外发起 HTTP 请求，企业内网可能需要配置代理。

问：如何迁移数据到另一台服务器？

答：直接复制 data/resourcehub.db 文件到目标服务器相同路径即可。若目标服务器数据库文件已存在且包含新数据，可使用 scripts/migrate.js 执行增量合并。请注意，SQLite 数据库文件与操作系统和 Node.js 版本无关，但建议两端的 Node.js 版本保持一致以避免 sqlite3 原生模块兼容性问题。
