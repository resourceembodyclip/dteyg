# WebLink Collective

WebLink Collective 是一个面向技术研究人员、内容聚合者与自动化采集系统的结构化外链资源汇总平台。项目定位于对半结构化移动端文章链接进行批量收录、分类标注与状态监控，解决分散链接难以统一管理、失效风险无法感知、应用场景不清晰等问题。本仓库作为第 54/56 批资源索引，收录 180 条来源于 h5.mobile.occzl.cn 的深度文章链接，并提供标准化的元数据描述、依赖环境声明、快速部署方案以及完整贡献流程，适用于需要长期维护大量外链目录的技术团队或个人开发者。

## 功能概览

**链接批量导入与去重**：支持从 CSV、JSON Lines 及纯文本列表中批量导入 URL，并基于完整的请求路径与查询参数进行精确去重，避免重复收录同一文章。

**多层级标签分类**：每条链接可关联领域、紧急程度、内容形态等自定义标签，便于按专题或业务线进行筛选与导出。

**可用性主动探测**：周期性对收录链接发起 HEAD 与 GET 请求，检测 HTTP 状态码、响应时间与内容哈希变化，标记异常链接并生成告警日志。

**结构化元数据扩展**：允许为每条链接补充标题、作者、发布时间、摘要、关联项目等字段，支持通过 YAML Front Matter 或附属 JSON 文件存储。

**外部引用关系追踪**：记录每条链接被本仓库内其他资源或外部站点引用的情况，生成引用网络图谱，辅助评估链接影响范围。

**静态站点生成适配**：内置模板引擎可将链接列表渲染为静态 HTML 目录页，方便直接部署到 Nginx 或对象存储服务，作为公开导航入口。

## 应用场景

**技术文档归档与交叉引用**：技术团队可将本仓库作为外部参考资料的统一入口，将分散在多个移动端页面中的技术分析、案例研究或 API 说明集中管理，并在内部 Wiki 或项目文档中直接引用本仓库生成的静态链接页。

**数据采集管道质量监控**：数据工程师可在定期采集任务完成后，将新抓取的链接追加至本仓库，并利用内置探测功能自动验证采集目标的可用性，快速定位源站改版或内容下架导致的采集失败问题。

**开源项目依赖外链审查**：开源项目维护者可将本仓库用于记录项目 README、文档站点或代码注释中出现的所有外部链接，便于定期审查外链的合规性与有效性，降低项目因引用失效链接而影响用户信任度的风险。

**内容推荐系统的候选集构建**：算法工程师可利用本仓库提供的分类标签与元数据，从大量移动端文章中按领域、热度或时间维度筛选候选内容，作为推荐系统召回阶段的种子链接池。

**个人知识库外链扩展**：独立研究者或写作者可通过本仓库维护个人阅读清单，将各渠道收集的移动端长文统一收录，配合标签与摘要字段构建可检索、可回溯的个人知识外挂系统。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需预先安装 Git、Node.js 18.x 及以上版本与 npm。

```bash
# 克隆仓库至本地
git clone https://github.com/your-org/weblink-collective.git
cd weblink-collective

# 安装项目依赖
npm install

# 执行链接索引构建与本地服务启动
npm run build
npm start
```

执行完毕后，本地服务默认监听 3000 端口，可通过 http://localhost:3000 访问生成的链接目录首页。若需更新链接列表，请将新 URL 按行追加至 `data/links.txt` 或 `data/links.json` 后再次执行 `npm run build`。

## 项目结构

```
weblink-collective/
├── bin/                                 # 可执行脚本目录
│   ├── cli.js                           # 命令行入口，暴露 build、check、serve 等子命令
│   └── cron-worker.js                   # 定时任务脚本，用于周期性探测链接状态（可配合 systemd 或 crontab）
├── config/                              # 项目配置目录
│   ├── default.yaml                     # 默认配置，含服务端口、探测超时、重试策略等
│   └── schema.json                      # 配置文件的 JSON Schema 校验定义
├── data/                                # 数据存储目录（不纳入版本控制，由初始化脚本创建）
│   ├── links.sqlite                     # SQLite 数据库文件，存储链接主表、标签表、探测记录表
│   ├── imports/                         # 外部导入暂存区，存放待处理的 CSV 或 JSON 批量文件
│   └── exports/                         # 导出生成目录，存放过滤后的链接列表或引用图谱报告
├── docs/                                # 项目文档
│   ├── user-guide.md                    # 面向最终用户的操作手册
│   ├── developer-guide.md               # 面向贡献者的架构与模块说明
│   ├── api-reference.md                 # 核心 API 的详细参数与返回值描述
│   └── operations.md                    # 生产环境部署与日常运维指南
├── src/                                 # 核心源码目录
│   ├── core/                            # 核心业务逻辑模块
│   │   ├── linker.js                    # 链接对象模型定义、校验与规范化
│   │   ├── importer.js                  # 从不同格式（文本、JSON、CSV）解析并导入链接
│   │   ├── probe.js                     # 链接可用性探测引擎，含状态码、响应时间、内容摘要计算
│   │   └── registry.js                  # 数据库访问层，封装对 links、tags、probes 表的 CRUD 操作
│   ├── render/                          # 渲染与输出模块
│   │   ├── static.js                    # 生成静态 HTML 目录页的核心逻辑
│   │   ├── graph.js                     # 生成链接引用关系 DOT 格式数据（可导入 Graphviz）
│   │   └── markdown.js                  # 将链接列表导出为 Markdown 表格或列表的工具函数
│   ├── utils/                           # 通用工具函数
│   │   ├── fetcher.js                   # 封装 axios 或 node-fetch，统一超时与重试
```

## 常见问题

**问：如何快速验证一批新链接是否全部可访问，而不必等定时任务触发？**

答：可使用命令行工具手动执行探测。在项目根目录运行 `npm run check -- --file data/imports/new-links.txt`，该命令会同步对指定文件中的链接发起探测请求，并将结果输出至终端表格。若需将结果持久化至数据库，可附加 `--save` 参数。

**问：数据库文件损坏或误删除后如何恢复？**

答：项目默认在 `data/backups/` 目录下保留最近 7 次自动备份（每次执行 `npm run build` 时触发）。若需恢复，请停止服务，将备份文件（如 `links-backup-20260615.sqlite`）复制覆盖 `data/links.sqlite`，然后重启服务。若备份也不可用，可重新运行 `npm run init-db` 初始化空数据库，然后通过 `data/imports/` 中的最新导出文件或原始链接列表重新导入。

**问：静态生成的 HTML 目录页如何部署到公网？**

答：执行 `npm run build` 后，静态文件输出至 `dist/` 目录。该目录包含 `index.html` 及配套的 CSS 与 JavaScript 资源。可直接将 `dist/` 整体上传至任意支持静态托管的服务，如 Nginx 根目录、Amazon S3 + CloudFront、Cloudflare Pages 或 Vercel。部署前建议修改 `config/default.yaml` 中的 `baseUrl` 字段为你的实际访问域名，以确保页面中的链接地址正确。
