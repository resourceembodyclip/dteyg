# WebLink Navigator

WebLink Navigator 是一个面向技术研究者和信息分析人员的结构化外链资源聚合与导航系统。该项目并非一个传统的内容管理系统或爬虫框架，而是一个精心编排的、可扩展的 URL 资源索引库，旨在将分散于各类技术文章、官方文档与行业报告中的高质量外链进行集中管理、分类标注与快速检索。

本项目主要服务于需要频繁查阅技术规范、跟踪行业动态、进行竞品分析或学术文献调研的用户群体。通过将数量庞大的原始链接纳入统一的目录体系，WebLink Navigator 有效解决了个人书签管理混乱、链接失效难以追溯、跨领域资源关联性弱等常见痛点。项目本身不存储任何文章内容，仅提供链接的元数据描述与结构化组织，确保轻量级、可移植且易于与现有工作流集成。

## 功能概览

**结构化链接索引** 对每一条收录的 URL 赋予唯一内部标识符，并按技术领域、内容类型与来源站点进行多维度分类，形成清晰的层级导航。

**元数据提取与标注** 自动或半自动地从链接目标页面的标题、关键词与首段文本中提取关键信息，生成摘要标签与内容概要，便于快速筛选。

**批量导入与去重** 支持从 CSV、JSON 或纯文本列表批量导入原始 URL，内置基于 URL 规范化的去重算法，避免重复收录，确保索引纯净度。

**多关键词检索** 提供基于标题、标签、描述字段的全文检索功能，支持布尔运算符与通配符查询，满足精细化检索需求。

**链接可用性健康检查** 周期性对已收录的 URL 发起 HEAD 请求，检测响应状态码，自动标记失效或重定向链接，生成可用性报告。

**分类视图与聚合展示** 按预设分类（如前端框架、后端中间件、数据库、云原生、学术论文库等）生成聚合视图，每个分类下展示最新添加或最常访问的资源。

## 应用场景

技术团队内部知识库构建：技术负责人可使用 WebLink Navigator 统一收集项目开发过程中参考的 API 文档、最佳实践博客、故障排查案例，形成团队共享的上下文资源池，减少重复搜索成本。

学术研究与文献综述：研究人员在撰写文献综述或技术报告时，可将大量散落在预印本平台、会议论文集与技术杂志中的相关链接统一收录，借助分类与检索功能快速定位特定主题的支撑材料。

技术资讯聚合与监测：开发者可将日常订阅的技术周报、行业分析网站、安全公告源链接纳入系统，通过健康检查功能及时发现失效的订阅源，确保信息渠道的持续有效。

产品竞品分析：产品经理与市场分析师可将竞品官方博客、版本发布日志、用户反馈论坛等链接汇总，通过元数据标注竞品功能迭代时间线，辅助决策。

## 快速开始

以下步骤将帮助您在本地环境中快速启动 WebLink Navigator 实例。

```bash
# 克隆项目仓库至本地
git clone https://github.com/weblink-navigator/weblink-navigator.git

# 进入项目根目录
cd weblink-navigator

# 安装项目依赖（使用 npm）
npm install

# 或者使用 yarn
yarn install

# 启动开发服务器，默认监听端口 3000
npm run dev

# 生产环境构建
npm run build
npm start
```

启动成功后，访问控制台输出的本地地址（通常为 http://localhost:3000 ）即可进入系统主界面。首次启动将自动初始化内置的 SQLite 数据库并创建默认分类与管理员账户，具体初始凭证请查看安装目录下的 .env.example 文件。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 16.20.0 及以上 | 项目运行时环境，建议使用 LTS 版本 |
| npm | 8.0.0 及以上 | Node.js 包管理器，用于安装项目依赖 |
| SQLite3 | 3.32.0 及以上 | 嵌入式关系型数据库，存储链接索引与元数据 |

| 可用磁盘空间 | 200 MB 及以上 | 用于存储代码、数据库文件及日志 |
| 网络连接 | 出站 HTTPS 访问 | 用于执行链接可用性健康检查与元数据抓取 |

## 常见问题

**问：导入大量 URL 时系统响应缓慢或超时，应如何优化？**

答：当一次性导入超过 500 条链接时，建议使用命令行脚本 `scripts/import-batch.js` 进行异步批量处理，而非通过 Web 界面上传。该脚本会将导入操作拆分为多个事务批次，每批 50 条，并输出详细的进度日志。此外，您可以在 `config/default.json` 中调整 `batchSize` 参数以匹配服务器性能。若导入源为 CSV 文件，请确保文件编码为 UTF-8 且包含必要的表头字段。

**问：链接健康检查报告显示大量超时或 403 状态，是否意味着系统故障？**

答：健康检查的超时或 403 状态并不直接指示系统故障。部分目标站点可能实施访问频率限制或 User-Agent 过滤策略。您可以修改 `src/services/healthCheckService.js` 中的请求头配置，将 User-Agent 设置为常见浏览器值。另外，检查超时阈值（默认 5000 毫秒）可根据网络环境在配置文件中调整。对于返回 403 的链接，系统会标记为“疑似受限”并跳过后续重试，避免被目标服务器封禁 IP。

**问：如何将当前 SQLite 数据库迁移至 PostgreSQL 或 MySQL 等生产级数据库？**

答：当前版本原生支持 SQLite，但项目架构已通过数据访问层抽象了底层实现。您可以通过修改 `config/production.json` 中的 `dialect` 参数并配置对应的连接字符串来切换至外部数据库。迁移时，请使用 `database/migrations/` 目录下的脚本作为 schema 参考，注意 SQLite 与其它数据库在数据类型（如布尔值、自增主键）上的差异。项目将在后续大版本中提供官方迁移辅助工具。
