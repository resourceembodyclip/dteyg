# ResourceBridge

ResourceBridge 是一个面向技术研发团队与个人开发者的外链资源归集与导航系统。该项目定位于解决多源技术文档、碎片化知识链接与内部参考资料难以统一管理、检索和共享的问题，通过结构化的元数据组织方式，将大批量外部 URL 转化为可检索、可分类、可审计的内部知识资产。ResourceBridge 不生产内容，但提供对原始内容的高效引用、状态监控与访问路由能力，适用于需要长期维护外部链接清单的中大型软件工程项目。

## 功能概览

- **海量链接归集**：支持单次导入上万条外部 URL，自动解析协议、域名、路径参数，并按来源站点与内容类型建立索引。
- **元数据增强**：为每条链接自动补充请求方法、状态码缓存、内容类型嗅探结果，支持手动标记业务领域、责任人、关联需求编号。
- **多维度检索**：基于域名、路径关键词、自定义标签、导入批次（如第 24/56 批）进行精确与模糊查询，支持正则表达式过滤。
- **访问健康巡检**：内置异步任务调度器，定期对已收录链接发起 HEAD/GET 请求，检测可达性、重定向链、SSL 证书有效期，输出异常报表。
- **权限与审计**：集成基于角色的访问控制（RBAC），记录每次链接查看、修改与删除操作的完整日志，满足企业内控要求。
- **导入导出接口**：提供 RESTful API 与命令行工具，支持 CSV/JSON 格式的批量导入导出，便于与外部数据源或 CI/CD 流水线集成。
- **分类视图模板**：预置技术文档、API 参考、运维手册、测试报告、设计提案等常见分类模板，可自定义分类层级。

## 应用场景

1. **技术文档中心迁移辅助**：团队在从 Wiki 迁移至 Confluence 或自建文档平台时，可使用 ResourceBridge 导出全部历史链接清单，进行死链检测与重定向映射，避免文档引用断裂。
2. **微服务依赖关系梳理**：架构师将各服务对应的配置中心、监控面板、日志查询入口等链接录入系统，按服务名称打标，形成统一的依赖拓扑参考页。
3. **外部情报聚合**：安全合规团队将厂商安全公告、CVE 数据库、补丁下载地址等外部链接统一托管，配合巡检功能监控官网变更，缩短漏洞响应时间。
4. **新人入职知识导航**：将内部常用开发环境、代码仓库、项目看板、审批系统等链接按角色分类生成快速访问目录，减少新员工问路式沟通成本。
5. **项目结项归档**：在项目收尾阶段，将所有用到的第三方库文档、协议规范、部署手册等链接整体打包导出，附带元数据快照，供后续审计或复盘使用。

## 快速开始

以下命令演示如何在本地环境克隆代码、安装依赖并启动开发服务器。

```bash
git clone https://github.com/your-org/resource-bridge.git
cd resource-bridge
npm install
npm run dev
```

如需生产环境构建，请执行 `npm run build` 并配合 `pm2` 或 `systemd` 进行进程守护。默认服务端口为 3000，可通过环境变量 `PORT` 覆盖。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x LTS 或 20.x LTS | 运行时环境，需支持 ES2022 特性 |
| npm | 9.x 或以上 | 包管理器，用于安装前端与后端依赖 |
| PostgreSQL | 14.x 或以上 | 主数据存储，需启用 JSONB 类型支持 |
| Redis | 6.2 或以上 | 缓存会话与巡检任务状态队列 |
| Nginx | 1.22 或以上 | 生产环境反向代理，承载静态资源与负载均衡 |
| Docker | 20.10 或以上 | 可选，用于容器化部署与开发环境一致性 |

## 项目结构

```
resource-bridge/
├── packages/                          # 多包管理目录
│   ├── backend/                       # 核心服务端（NestJS + TypeORM）
│   │   ├── src/
│   │   │   ├── modules/               # 功能模块：链接、巡检、审计、分类
│   │   │   ├── scheduler/             # 基于Bull的巡检任务队列
│   │   │   └── common/                # 全局过滤器、拦截器、中间件
│   ├── frontend/                      # 管理控制台（React + Vite）
│   │   ├── src/
│   │   │   ├── pages/                 # 链接列表、详情、巡检报表、权限管理
│   │   │   ├── components/            # 可复用UI组件（DataTable、TagInput）
│   │   │   └── hooks/                 # 自定义数据请求与缓存钩子
│   └── cli/                           # 命令行工具（Commander + Axios）
│       ├── commands/                  # import、export、health-check 子命令
│       └── utils/                     # CSV解析、进度条、日志格式化
├── docker/                            # 容器化部署资源
│   ├── docker-compose.yml             # 整合PostgreSQL、Redis、Nginx
│   └── Dockerfile.backend             # 多阶段构建生产镜像
├── docs/                              # 完整文档（用户手册、API、运维）
│   ├── user-guide.md
│   ├── admin-guide.md
│   ├── api-reference.md
│   └── contributing.md
├── scripts/                           # 辅助运维脚本
│   ├── init-db.sql                    # 数据库初始化表结构与索引
│   └── seed-links.js                  # 批量导入示例数据（含本批次链接）
├── tests/                             # 单元测试与集成测试（Jest + Supertest）
│   ├── unit/                          # 服务层与工具函数单测
│   └── e2e/                           # 接口全链路测试
├── .env.example                       # 环境变量模板（数据库连接、Redis地址）
├── package.json                       # 根目录依赖与workspace配置
└── README.md                          # 项目说明文档（当前文件）
```

## 贡献指南

1. **问题跟踪**：请在 GitHub Issues 中查找现有任务或提交新缺陷，描述时附上复现步骤、日志截图以及受影响链接的ID范围。涉及安全相关问题请直接发送邮件至 security@resource-bridge.internal。

2. **分支规范**：基于 `develop` 分支创建功能分支，命名格式为 `feature/描述` 或 `fix/描述`。所有提交必须通过 ESLint 与 Prettier 检查，并包含至少一个单元测试用例。

3. **代码评审**：发起 Pull Request 时需填写变更摘要、测试覆盖说明以及影响面评估。至少需要一名核心维护者批准方可合并。CI 流水线会自动执行构建、测试与代码覆盖率检查，覆盖率不得低于 80%。

4. **文档同步**：任何对外接口变更（API 字段、配置项、数据库表结构）必须同步更新 `/docs` 目录下的对应文档。新功能需在 `user-guide.md` 中增加操作说明，并提供命令行或界面截图示例。

5. **发布流程**：维护者按语义化版本（SemVer）规则定期发布 Release，更新 `CHANGELOG.md`，并在 Release Notes 中列出重大变更、废弃特性及迁移建议。

## 常见问题

**Q：导入大量链接时出现超时或内存溢出，如何处理？**

A：推荐使用 CLI 工具的 `--batch-size` 参数将导入任务分片，例如每 500 条提交一次事务。同时确保 PostgreSQL 的 `work_mem` 和 `shared_buffers` 根据机器配置适当调高。若仍需优化，可启用 Redis 队列将导入过程改为异步后台任务。

**Q：巡检任务报告显示大量链接状态码为 403 或 429，是否视为异常？**

A：部分外部站点会对频繁请求做限流或验证拦截。ResourceBridge 默认使用 `User-Agent: ResourceBridge-HealthCheck/1.0` 并支持配置 `rate-limit-delay` 参数（单位毫秒）来降低请求频率。若目标站点明确要求白名单机制，建议在巡检配置中跳过该域名或使用代理轮询。

**Q：如何从旧版本迁移到当前最新版本？**

A：每个 Release 版本都会附带 `migration-guide.md` 文档，其中包含 SQL 变更脚本、新增配置项默认值以及兼容性说明。通常流程为：备份数据库 -> 执行增量迁移脚本 -> 更新 `.env` 文件 -> 重启服务。若跨大版本升级，建议先在预发布环境演练一遍。
