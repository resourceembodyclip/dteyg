# OCCZL Resource Aggregator

OCCZL Resource Aggregator 是一个面向技术研究人员、数据分析师和内容管理者的外链资源汇总工具，专注于从移动端与桌面端双渠道采集、组织和呈现来自 occzl.cn 域下的高质量技术文章与资讯内容。该项目不生产内容，而是通过结构化的方式将散落在各文章ID下的技术文档、案例分析和行业报告进行统一索引，帮助用户快速定位特定编号的文章资源，避免在海量书签中反复查找。

项目定位为轻量级的技术资源导航站，适用于需要频繁查阅特定编号文章的技术团队、需要批量引用外部资源的文档编写者，以及希望建立个人知识库外链映射的研究人员。通过本项目提供的资源清单，用户无需记忆复杂的文章ID，即可按类别、按批次获取完整的文章入口列表，显著提升信息检索效率。

## 功能概览

**多协议资源接入** 同时支持 http 与 https 协议下的文章链接采集，覆盖 m.mobile 子域和 www 子域的双通道数据源，确保资源获取的完整性与兼容性。

**文章ID批量管理** 以文章数字ID为核心索引键，对每篇文章进行唯一标识，支持按ID范围、按批次、按入库时间进行筛选与导出，便于大规模资源治理。

**移动端与桌面端分离展示** 将 mobile 子域和 www 子域的资源分别归类，适配不同终端场景下的访问习惯，移动端链接自动识别为轻量页面，桌面端链接保留完整布局。

**原始链接透传** 所有资源链接均以原始URL形式原样呈现，不进行任何协议升级、域名改写或路径归一化，确保链接的有效性与可追溯性。

**ASCII目录树可视化** 项目内置完整的文件结构说明，帮助贡献者与使用者快速理解代码组织方式，降低协作门槛。

**跨批次资源编排** 本项目属于第14/56批资源处理计划，共整合180个外链资源，提供批次进度追踪与批次间交叉引用功能。

**无依赖轻量化运行** 项目核心为静态资源列表，无需额外安装数据库或后端服务，纯前端可访问，适合部署于任何静态托管平台。

## 应用场景

技术文档团队内部知识库建设
技术写作团队需要频繁引用外部文章作为参考依据，但文章URL分散且编号无规律。通过本项目提供的按ID排序的资源清单，团队成员可在本地或内网环境中快速检索目标文章ID，直接跳转原文，减少在浏览器历史记录中的翻找时间。

数据分析师批量导出文章元数据
数据分析师需要统计 occzl.cn 域名下的文章发布规律、ID分布密度或协议使用比例。本项目提供完整的文章URL集合，分析师可将其导入爬虫框架或Excel进行二次加工，生成词频图、时间序列图等可视化报表。

个人开发者构建外链监控系统
个人开发者可利用本项目提供的资源列表作为基础数据源，编写定时脚本检测每篇文章的可访问性、响应时间或状态码变化，从而构建一个轻量级的外链健康监控看板，及时发现失效链接。

内容运营人员策划专题合集
内容运营人员需要围绕特定技术主题策划文章合集，但缺乏统一的文章索引。本项目按批次聚合了180篇文章入口，运营人员可按ID段快速浏览文章标题（需访问原文），筛选出匹配主题的文章进行专题包装。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/your-org/occzl-resource-aggregator.git

# 进入项目目录
cd occzl-resource-aggregator

# 安装依赖（项目为纯静态资源，仅需安装开发服务器依赖）
npm install -g serve

# 启动本地开发服务器
serve -s . -p 3000
```

执行上述命令后，在浏览器中访问 http://localhost:3000 即可查看资源列表首页。若需更新资源清单，直接编辑 `data/resources.json` 文件并按JSON格式追加新条目，保存后刷新页面即可生效。

## 贡献指南

1. 提交新资源条目
在 `data/resources.json` 文件中按照 `schema.json` 定义的格式追加新对象，必须包含 `id`（数字类型）、`url`（字符串类型）和 `subdomain`（枚举值：mobile 或 www）三个字段。提交前运行 `npm run validate` 确保数据格式合法。

2. 更新批次元信息
当新增资源数量累计达到50条或完成一个完整批次时，需在 `data/batch-meta.json` 中更新 `currentBatch` 和 `totalResources` 字段，并记录 `lastUpdated` 时间戳。本项目当前为第14批次，共180个资源。

3. 修改UI样式或交互逻辑
若需调整前端页面布局或交互行为，请先在 `css/main.css` 或 `js/app.js` 中修改对应代码，然后执行 `npm run test` 运行全部单元测试与集成测试，确保无回归问题。提交PR时附上修改前后的截图对比。

4. 编写或更新文档

## 常见问题

Q: 为什么部分文章ID在列表中重复出现？

A: 由于数据源来自不同子域（m.mobile 与 www）且文章ID可能在不同子域下对应不同内容，因此重复ID是允许的。项目设计上以完整URL作为唯一标识，而非仅依赖数字ID。若确认为同一篇文章的重复收录，可通过去重脚本（位于 `scripts/deduplicate.js`）进行清理，但需谨慎操作以避免误删有效链接。

Q: 如何快速检查这批180个链接中哪些已经失效？

A: 项目根目录下提供了 `scripts/check-health.sh` 脚本，利用curl命令并行检测每个链接的HTTP状态码。执行 `./scripts/check-health.sh` 后，脚本会生成 `health-report.csv` 文件，其中包含每个URL的响应状态、响应时间和重定向链路，方便您识别异常链接。

Q: 我可以通过API方式获取这批资源列表吗？

A: 可以。项目内置了一个简易的JSON数据服务，启动本地服务器后，访问 `http://localhost:3000/api/resources` 即可获取完整的资源列表JSON响应，支持 `?subdomain=mobile` 和 `?minId=1000&maxId=9999` 等查询参数进行过滤。该接口可用于与其他系统集成或构建自动化工具。
